# 오늘 한 일

- Indy7 로봇 컨트롤러 네트워크 연결 (USB 이더넷 동글 고정 IP 설정)
- ROS2 Humble 워크스페이스 `colcon build` 성공
- `/dev/ttyACM0` 권한 문제 해결 → Mark7 그리퍼 정상 구동
- Direct teaching(핸드가이드) ON/OFF 기능 Indy driver에 패치

# 왜 이 작업을 했는가

로봇과 PC 간 통신이 안 되면 아무것도 할 수 없다. 네트워크·빌드·권한 문제를 한 번에 정리해 데이터 수집 환경을 완성하기 위해.

# 작업 과정

**네트워크:** `ping 192.168.1.10` 100% loss → USB 동글 인터페이스(`enx00e04c360046`)에 고정 IP가 없었음 → `nmcli con add`로 `indy7-static` connection 생성 후 연결 성공.

**colcon build:** `catkin_pkg`, `empy(3.3.4)`, `numpy`, `lark-parser` 순서대로 의존성 누락 발생 → 각각 pip 설치로 해결. conda python(`/miniconda3/bin/python3`)이 빌드에 관여하는 것이 핵심이었음.

**Mark7 권한:** `/dev/ttyACM0`가 `dialout` 그룹 소유 → `sudo usermod -aG dialout $USER` 후 `sg dialout`로 즉시 우회 적용.

**Direct teaching:** `indy_driver`가 `data=9/10`을 처리하지 않았음 → `indy_define.py`에 상수 추가, `indy_driver.py`에 `set_direct_teaching()` 호출 로직 패치.

# 중간 결과

- `ping 192.168.1.10` 성공 (2/2 패킷)
- `colcon build` 전 패키지 정상 완료
- `data_collection.launch.py` + `system_teleop_node` 정상 실행
- `D`(Direct teaching ON), `SPACE`(녹화 시작/중지), `Y/N` 라벨링 동작 확인

# 막힌 점

- `nmcli con mod`로 수정 시도했으나 connection 이름이 없어 실패 → `con add`로 새로 생성해야 했음
- `empy` 버전이 4.x면 `em.BUFFERED_OPT` 누락으로 빌드 실패 → 3.3.4로 다운그레이드 필요
- `newgrp dialout` 후에도 세션 그룹이 바로 반영 안 됨 → `sg dialout` 명령으로 우회

# 느낀 점

환경 구축이 이렇게 힘든데 실제 AI는 언제 하나 싶었다. 하지만 하나씩 해결되니까 뿌듯함도 있음. conda와 시스템 python이 섞이면 반드시 어떤 python이 빌드에 쓰이는지 먼저 확인해야 한다는 교훈.
