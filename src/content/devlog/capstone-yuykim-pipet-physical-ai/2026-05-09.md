# 오늘 한 일

- 100k 모델 실기 추론 결과 분석
- Gemini Robotics-ER를 grasp gate 검증기로 실험
- grasp-focus NPZ 데이터 생성 (200 에피소드)
- LeRobot 변환 완료

# 왜 이 작업을 했는가

100k 모델이 파이펫에 접근하긴 하지만 마지막 2cm에서 흔들리고 grasp가 너무 빨리 나온다. 코드 보정만으로는 한계 → 데이터 보강 필요.

# 작업 과정

**실기 문제 재정의:** 파이펫 미검출이 아니라 "마지막 2cm 정밀 접근 불안정 + grasp 타이밍 부정확".

**Gemini-ER 실험:**
- `gemini-robotics-er-1.6-preview` API 호출 성공
- wrist/overhead 이미지 기반 파이펫 위치/grasp 타이밍 판단 시도
- 결과: HTTP 429 quota/rate-limit 반복 → 실시간 gate로 쓰기 어려움
- 결론: 오프라인 분석에는 유용, 실시간은 데이터 보강이 우선

**grasp-focus 데이터 생성:**
- `make_grasp_focus_episodes.py` 스크립트 추가
- 원본 100ep에서 첫 grasp 기준 ±4.0s/2.0s window crop → 100 focus ep 추가
- 결과: 200 NPZ, ~37GB
- LeRobot 변환: `pipet_extended_depth_100_grasp_focus` (47,222 frames)

# 중간 결과

- grasp-focus 데이터셋 완성
- 60k 스텝 학습 계획 수립 (기존 100k 모델과 비교용)

# 막힌 점

- idle frame 삭제 때문에 grasp 구간이 빠진 게 아닌지 의심했으나 → raw transition 수와 변환 후 frame 수가 일치, 아니었음
- Gemini API quota가 너무 낮음 (0.5Hz로도 429 발생)

# 느낀 점

VLM 보조 판단기 아이디어는 매력적이지만 API quota 현실이 있다. 당장은 더 좋은 데이터가 답이다.

# 다음 할 일

- grasp-focus 데이터로 60k 학습 실행
- 기존 100k 모델과 실기 비교
