# 오늘 한 일

- 팀 레포에서 개인 fork로 분기된 `dogyung`, `movetelel`, `yuykim` 브랜치를 `main`으로 통합
- 디렉토리 구조 1차·2차 리팩토링
- `STRUCTURE.md` 추가
- 블로그 자동 배포 설정 (`.devlog.yml`, `publish-devlog.yml`)

# 왜 이 작업을 했는가

브랜치가 3개로 분산되어 있어서 어느 코드가 최신인지 불명확. 디렉토리도 의미 없이 섞여 있어 `control_indy7`이 외부 코드인지 내부 코드인지 알 수 없었음.

# 작업 과정

**브랜치 통합:**
- `yuykim`이 이미 `movetelel`을 머지한 상태
- `dogyung`도 반영 완료 → 충돌 없이 fast-forward
- 원격에서 `dogyung`, `movetelel`, `yuykim` 브랜치 삭제, `main` 하나만 유지

**1차 정리:**
- `.taskmaster/` 제거
- `external/control_indy7/` 삭제 (루트 버전으로 통합)
- `ai/indy7_lerobot/` 빈 폴더 제거
- `ToDokyung/` → `docs/progress/`
- 루트 `wrist_*.png` → `img/`

**2차 리팩토링:**
- `control_indy7/` → `tools/control_indy7/` (1st-party 운용 툴)
- `external/pipet_gripper_Mark7/` → `vendor/pipet_gripper_Mark7/`
- `mujoco_env/indy7_urdf/`에서 불필요한 565개 파일 삭제 (타 로봇 모델, `.meta`, prefab 등)
- 깨진 경로 참조 수정: `operator_gui.py`, `prepare_models.py`, `mujoco_env/README.md`

**블로그 연결:**
- `.devlog.yml`, `.github/workflows/publish-devlog.yml` 추가
- `history.md` `docs/` → 루트로 이동

# 중간 결과

- `main` 브랜치 하나로 정리 완료
- `tools/`, `vendor/` 의도가 명확한 구조
- `STRUCTURE.md`로 디렉토리 역할 문서화
- 블로그 자동 배포 파이프라인 연결

# 막힌 점

- `control_indy7`을 `external/`로 옮겼다가 `operator_gui.py` 경로 깨짐 → 2차에서 `tools/`로 재분류 및 참조 수정
- `mujoco_env/indy7_urdf/`가 Unity 에셋 전체(~580파일)를 담고 있어서 정리에 시간이 걸림
- 블로그 배포: `receive-devlog.yml`이 GITHUB_TOKEN 커밋 사용 → `deploy.yml` 미트리거 이슈 발견 (별도 수정)

# 느낀 점

레포 정리는 미루면 미룰수록 더 힘들어진다. 의도별로 폴더를 나누고 나니 "이게 뭐하는 코드지?"라는 질문이 줄었다. 블로그 연결도 해서 앞으로 기록이 자동으로 쌓이게 됐다.
