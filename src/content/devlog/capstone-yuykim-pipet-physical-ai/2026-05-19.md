# 오늘 한 일

- grasp-focus v2 학습 계획 수립 (lr=3e-5, cosine scheduler, dropout=0.05)
- DAgger correction 수집 파이프라인 (`pipet_dagger_collection` 패키지) 추가
- DAgger 기본 속도 보수화 및 연속 수집 UX 개선

# 왜 이 작업을 했는가

기존 grasp-focus 학습이 50k 이후 loss 평탄 → lr, scheduler 개선 필요. 또한 모델이 실패하는 구간에서 사람이 직접 개입한 correction 데이터만 모으면 효율적인 성능 개선 가능.

# 작업 과정

**grasp-focus v2 학습:**
- `adamw, lr=3e-5, weight_decay=1e-4, grad_clip_norm=10`
- `cosine_decay_with_warmup`: warmup=2000, decay=80000, peak=3e-5, decay_lr=3e-6
- `dropout=0.05` (기존 0.1에서 완화)
- `drop_idle_sec=1.0, idle_keep_action_window_sec=2.0` (정지 구간 제거, grasp 보호)
- 목표 loss: 0.12~0.15 구간

**DAgger 파이프라인:**
- 새 패키지 `pipet_dagger_collection` (기존 코드 수정 없음)
- `dagger_supervisor_node`: 평소 모델 출력을 `/indy/teleop_pose`로 forwarding
- 사람이 START → 수집 시작, 종료 후 A/B/X로 라벨링
- 저장 경로: `episodes/remove_dagger/<label>`

**UX 개선:**
- 저장/폐기 후 HOME 복귀 → AUTO rollout 재개 (매 episode 재실행 불필요)
- Pygame UI에 `control:` 상태 표시
- 왼쪽/오른쪽 스틱 클릭으로 step 조절

**속도 보수화:**
- `max_cartesian_speed_mm_s: 8.0 → 2.0`
- `max_delta_mm: 1.0 → 0.5`
- joystick deadzone: `0.18 → 0.22`

# 중간 결과

- DAgger supervisor, supervisor node 코드 완성
- 연속 수집 UX (HOME → AUTO → TAKEOVER 사이클) 구현

# 막힌 점

- AUTO forwarding 전에 `indy_srv` 서비스 대기로 토픽 publish가 막히는 현상 → inference launch가 이미 teleop 준비하므로 AUTO에서는 바로 forwarding하도록 수정
- BACK emergency stop과 BACK+LB/RB 조합이 충돌 → 주 조작법에서 BACK+LB/RB 제외

# 느낀 점

DAgger 아이디어 자체가 흥미롭다. 모델이 틀리는 순간만 골라서 사람이 시범을 보이면 데이터 효율이 훨씬 높다. UX를 잘 만들어야 실제로 쓸 수 있다는 걸 느꼈다.
