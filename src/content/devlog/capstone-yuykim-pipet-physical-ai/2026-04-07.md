# 오늘 한 일

- ACT 학습 설정 변경 (`chunk_size=40`, `use_amp=true`, `image_resize_to 360x480`)
- NPZ → LeRobot v3 변환 + 첫 학습 실행
- 디스크 부족·스트리밍 OOM 등 파이프라인 문제 해결

# 왜 이 작업을 했는가

기존 학습이 chunk_size=100으로 너무 컸고, 이미지 해상도도 원본 그대로라 불필요하게 무거웠다. 설정을 최적화하면서 실제 학습을 돌려보기 위해.

# 작업 과정

**학습 설정 변경:**
- `policy.use_vae=false`, `chunk_size=40`, `n_action_steps=40`, `use_amp=true`
- 이미지 기본 해상도: `360x480`

**파이프라인 트러블슈팅:**
1. `FileExistsError` - LeRobot이 기존 출력 디렉토리 있으면 실패 → 재실행 시 `rm -rf` 후 진행
2. `No space left on device` - 루트 파티션 여유 2GB → HF 캐시를 `ai/.cache/huggingface`로 이동, 15GB+ 확보
3. 스트리밍 학습 시도 → OOM + `ValueError: Batch does not contain any data` → 비스트리밍으로 확정
4. `log_freq` 기본 200이라 한참 조용 → 50으로 단축

# 중간 결과

- 비스트리밍 + `num_workers=2` + `HF_HOME` 재설정으로 학습 정상 시작
- `Training: ... step` + loss 하강 확인
- `ai/lerobot/train_26_04_07.sh` 스크립트로 재현 가능하게 정리

# 막힌 점

- 스트리밍 경로의 `_get_delta_frames`에서 에피소드 인덱스가 0차원 텐서여서 `RuntimeError` → `streaming_dataset.py` 패치 (스트리밍 비추천)
- conda `editable` 설치 경로가 `ai/huggingFace/lerobot`(불완전 복사본)을 가리켜 `ModuleNotFoundError` → `ai/lerobot_source/lerobot`으로 재설치

# 느낀 점

파이프라인을 처음 구축할 때는 항상 이런 잡다한 문제들이 쏟아진다. 하나씩 기록해두면 나중에 같은 문제를 반복하지 않는다.
