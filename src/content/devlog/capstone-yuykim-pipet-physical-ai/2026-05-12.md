# 오늘 한 일

- `act_pipet_extended_depth_100_grasp_focus` 080000 체크포인트 실기 10회 테스트
- grasp 로그 분석 및 원인 파악
- early-grasp 방지 파라미터 3개 추가

# 왜 이 작업을 했는가

grasp-focus 데이터로 학습한 80k 모델이 기존보다 나아졌는지 확인하고, 여전히 남은 "너무 빨리 잡기" 문제를 코드로 보정하기 위해.

# 작업 과정

**실기 결과:**
- 10회 중 3회 성공, 7회 실패
- 파이펫이 가까울 때 성공 가능성 높음
- grasp-focus 효과는 분명히 있음 (70k 대비 개선)
- 핵심 실패 패턴: "너무 빨리 잡기" (파이펫이 충분히 들어가기 전에 손가락 닫힘)

**로그 분석:**
```
grasp gate 통과: confirm=8/8, delta_norm=0.0006
grip_preset_node: grasp: [0.0, 0.0, 350.0, ...]
```
→ 하드웨어 오동작이 아니라, 모델이 잘못된 위치에서도 `grasp`를 출력하는 것이 문제

**early-grasp 방지 파라미터 추가:**
- `grasp_min_elapsed_steps=50` (추론 시작 후 ~2.5초 grasp 보류)
- `grasp_min_motion_rad=0.015` (일정량 이상 움직이기 전 grasp 보류)
- `enable_gripper=false` (디버깅용, 팔만 움직이고 그리퍼 차단)

# 중간 결과

- 추천 시작값: `grasp_delay_steps=12`, `grasp_confirm_steps=12`, `grasp_max_delta_norm=0.006`, `grasp_min_elapsed_steps=50`
- Python AST parse + ROS build 통과

# 막힌 점

- `use_zmq_sidecar:=true`일 때 실제 모델은 ROS launch의 `model_path`가 아니라 ZMQ 서버 쪽 경로 → ZMQ 서버 시작 로그로 경로 확인 필요

# 느낀 점

3/10은 아직 실용적이지 않다. 하지만 grasp-focus 데이터 효과가 분명히 있다는 게 확인됐다. 더 나은 데이터 수집 방법론이 필요하다.
